# Build Prompt: "Daily" — a personalized daily-puzzle platform powered by Gemini

> Paste this whole document into your coding agent (Claude Code, Cursor, etc.) as the project brief. It is written as instructions to the agent that will build the app. Read the **Non-negotiable architecture** and **How this stays feasible** sections before writing any code — they are the spine of the whole project.

---

## 1. Mission

Build a web app (mobile-first, responsive) that offers a large collection of daily puzzle games in the spirit of the "-dle" family (Wordle, LoLdle-style guessing games), the LinkedIn puzzle suite (Queens, Tango, Pinpoint, Crossclimb, Zip, Mini Sudoku, word-weaves), and the broader daily-puzzle web ecosystem. The app must:

1. **Onboard** each new user with a short interest picker, then serve them a personalized feed of games based on those interests.
2. Offer two modes for every game type:
   - **Daily mode** — one single canonical puzzle per *(topic × game type)* per day, **identical for every player in the world**, with a **global daily leaderboard** and **per-user statistics/streaks**.
   - **Infinite mode** — an effectively endless supply of fresh, valid puzzles in the same formats, for replayability with no risk of running out in a day.
3. Let players **skip** the current puzzle or swap topics freely so they never get stuck or bored.
4. Use the **Gemini API** as the engine that generates and curates the puzzle content **in the background**, not on the user's request path.

The product feeling to aim for: "open the app, get a stack of bite-sized puzzles tuned to things I actually care about, compete on today's boards, and if I want more I can play forever."

---

## 2. Non-negotiable architecture (read this first)

These constraints exist because of hard requirements (a worldwide-identical daily puzzle, LLM unreliability, latency, and cost). Do not violate them.

1. **Separate generation from serving.** Content is *generated and validated offline/in the background*, written to a database "content pool," and only then *served* to players. The request path that a player hits is fast, deterministic, and never calls Gemini live.
2. **Never serve unvalidated LLM output.** Every generated puzzle passes through a **validation gate** before it can enter the pool. If it fails (unsolvable, ambiguous, factually wrong, duplicate, off-format), it is discarded and regenerated. Treat all raw Gemini output as untrusted.
3. **Logic puzzles are generated by real solvers, not by the LLM.** Grid-logic games (Sudoku, Queens, Tango/binary, Zip-style path, word ladders) must be produced by deterministic generators/constraint-solvers that *prove* a unique solution. The LLM may pick difficulty/size/theme parameters, but it must never be the source of the puzzle structure — LLMs do not reliably produce uniquely-solvable logic puzzles.
4. **The daily puzzle is chosen once, by a date-derived seed, and frozen.** A nightly job selects/finalizes the day's canonical puzzle for each *(topic × game type)* and stores it immutably. Every player on that date gets that exact stored puzzle. No per-user generation in daily mode, ever.
5. **Build engines, not clones.** Implement a small number of **parameterized game engines** (archetypes), each driven by a *topic dataset*. Do not hand-build hundreds of one-off games, and do not clone specific branded games asset-for-asset (see §10 on IP). "LoLdle" and "Pokédle" are *mechanical inspiration* for the attribute-deduction engine — not products to reproduce with their copyrighted characters.

## 3. How this stays feasible (the part to get right)

This directly answers "how do we make Gemini-generated games realistic at scale?"

- **Amortized cost, not per-play cost.** Generation happens in **batched background jobs**, shared across all players. A player pulling their 30th infinite puzzle triggers zero Gemini calls — they pull a pre-built row from the pool. Cost scales with *content produced*, not *plays served*.
- **A buffer worker with a low-water-mark.** For each *(topic × engine)*, maintain a target pool depth (e.g. keep ≥ 200 unplayed infinite puzzles available). A background worker wakes on a schedule, checks which pools are below their low-water-mark, and refills them via Gemini batch jobs. This is what guarantees "can't run out in a day."
- **Datasets are curated once and reused.** For attribute-deduction games, Gemini's job is to *build and validate a topic dataset* (e.g. 200 entities with clean attributes) — done occasionally, reviewed, then cached. After that, generating "today's puzzle" is just a deterministic seeded pick from the validated dataset and needs **no LLM call at all**. Most daily puzzles cost nothing per day to produce.
- **Model tiering.** Use a fast, cheap model tier for high-volume bulk generation and a stronger model tier for harder curation and for the validation/judge pass. (Confirm current Gemini model IDs and the Batch API in Google's docs — model names change; do not hardcode assumptions.)
- **Aggressive caching + dedup.** Hash every generated artifact; reject near-duplicates so the pool stays varied. Cache served daily puzzles in a fast store (e.g. Redis) keyed by date+topic+engine.
- **Validation makes cheap models safe.** Because a deterministic gate verifies solvability/fairness/factual grounding, you can lean on cheaper models for volume without shipping broken puzzles.

Concrete scale target to design for: ~**15–25 topics** × ~**8 engines**. Daily mode then yields **100+ distinct global daily puzzles**; infinite mode draws from pools of **hundreds to thousands** per topic, continuously refilled. That is far more than any single player exhausts in a day.

## 4. The game engines (archetypes)

Implement each engine as a self-contained module with: a `generate(topicDataset, difficulty, seed)` producer, a `validate(puzzle)` gate, a `score(attempt)` function, a play UI, and a `dailySeedPick(date, topic)` selector. Engines are reused across all topics.

**A. Word Guess (Wordle-style).** Guess a hidden word in N attempts with per-letter feedback (correct / present / absent).
- *Generate:* Gemini produces a topical word list (e.g. animals, tech terms, country names); filter every candidate against a real dictionary/allowlist.
- *Validate:* answer and all accepted guesses exist in the dictionary; consistent length; not previously used recently.

**B. Attribute Deduction ("Classic" LoLdle/Pokédle-style).** Guess a hidden entity from a topic set; after each guess, reveal attribute comparisons (exact match / partial / none, and higher/lower for numeric fields like year).
- *Generate:* the heavy lifting is a **validated entity dataset per topic** (entities + attribute columns). Build it with Gemini grounded against open/authoritative sources, then validate. The daily puzzle is a seeded pick from this set.
- *Validate:* every attribute present and type-correct; values grounded in the source dataset; answer is in the set. This engine benefits most from the interest picker — pick "Football clubs" or "Programming languages" and you get a tuned entity set.

**C. Association & Grouping (Pinpoint + Connections-style).** Two variants: (i) reveal up to 5 clue words one at a time; guess the connecting category in as few reveals as possible. (ii) Sort 16 items into 4 hidden groups.
- *Generate:* Gemini (creative-language task it's genuinely good at) proposes clue sets / groupings, plus a second **LLM-as-judge** pass for ambiguity.
- *Validate:* every clue genuinely fits the category; no clue fits a *different* obvious category better; for the grouping variant, exactly one valid partition with no overlap traps.

**D. Grid Logic (Queens / Tango-binary / Mini-Sudoku / Zip-path).** Pure logic puzzles on a grid; topic-agnostic.
- *Generate:* **deterministic generators / constraint solvers only.** No LLM in the structure. The LLM may at most pick size/difficulty.
- *Validate:* solver proves a **unique** solution; reject anything with 0 or >1 solutions; grade difficulty by solve technique depth.

**E. Word Ladder (Crossclimb-style).** Transform between words changing one letter per rung; then order the rungs; bonus end-cap words.
- *Generate:* graph search over the dictionary for valid one-letter-step ladders; Gemini writes the per-rung clues.
- *Validate:* each step differs by exactly one letter and is a real word; each clue resolves to exactly one rung word.

**F. Word Path / Weave (Strands / Wend-style).** Find themed hidden words by tracing connected tiles through a letter grid; use tiles per the rules.
- *Generate:* deterministic word-packer places a Gemini-produced themed word set into the grid.
- *Validate:* every target word is traceable along connected tiles; grid constraints (tile usage, blocked cells) satisfied; theme coherent.

**G. Progressive-Clue Trivia (Gamedle-style "more clues each miss").** Guess an answer; each wrong guess unlocks an additional **text or emoji** clue. Deliberately uses text/emoji, **not** copyrighted audio or images (see §10).
- *Generate:* Gemini grounded trivia + an ordered clue chain (vague → specific), optionally an emoji-rebus clue.
- *Validate:* facts grounded against a source; clue ordering monotonically helpful; answer unambiguous.

**H. Proximity / Higher-Lower (Globle / higher-lower-style).** Guess toward a target using "closer/further" or "higher/lower" feedback on an open-data numeric attribute (population, area, release year, etc.).
- *Generate:* open datasets curated by Gemini.
- *Validate:* all numeric values come from an authoritative/open source; ranges sane.

> Note: media-clip games (song-snippet "Heardle"-style, movie-still "Framed"-style) are intentionally **excluded** because of media copyright; engine **G** covers the same "progressive reveal" fun using text/emoji clues instead.

For breadth inspiration on formats (mechanics only — do not copy branded content/IP), the agent may consult the public curated list at `github.com/aukspot/dles`.

## 5. Topics & onboarding

- **Onboarding flow:** on first launch, show a friendly interest picker — a grid of tappable topic chips grouped into families (e.g. *Geography, Science & Tech, Sports, Pop culture & media, Music, History, Language & words, Logic & numbers, Food*). Let the user pick several and set a difficulty preference. Allow "surprise me." Store the selections to the user profile; let them edit anytime in settings.
- **Topic → engine mapping:** each topic declares which engines it supports and points to its dataset(s). "World Geography" might power Attribute-Deduction (countries), Proximity (Globle-like), Word Guess (capital cities), and Trivia. Logic engines (D) are topic-agnostic and always available.
- **Gemini-assisted topic expansion:** let Gemini *suggest* new topics from a user's stated interests ("you like Formula 1 → try: F1 circuits, F1 drivers, motorsport history") and **propose new topic datasets** for an admin to approve. New approved topics feed the same generation pipeline. Keep a human-approval step before a new topic goes live.
- The personalized **feed** orders the user's games by their topic affinity, mixing daily cards (with today's leaderboard teaser) and an "infinite" entry point per favored topic.

## 6. Gemini integration design

**Where Gemini IS used (background only):**
- Building & enriching topic **datasets** for engines B and H (grounded, then validated).
- Producing topical **word lists** for engines A, E, F (then dictionary-filtered).
- Authoring **clues / categories / groupings / trivia** for engines C, E, G.
- The **validation/judge pass** (LLM-as-judge for ambiguity/fairness, complementing deterministic checks).
- **Flavor**: hints, success messages, daily theme names, topic suggestions from user interests.

**Where Gemini is NOT used:**
- The structure of logic puzzles (engine D) and the legality of word ladders/weaves (E, F) — deterministic solvers own correctness.
- The live request path — serving, scoring, leaderboards, stats are pure backend.
- Final admission to the pool — that's the deterministic gate.

**Implementation requirements:**
- Use the **official Google Gen AI SDK**; call from a **server-side worker only**. The API key lives in server env/secret storage and is **never shipped to the client**.
- Use **structured output** (JSON response schema) for every generation call so results parse deterministically; reject and retry on schema violations.
- Use the **Batch API** for bulk generation to cut cost/latency where async is acceptable; use grounding-with-search for factual datasets/trivia.
- Wrap calls with **retries, timeouts, rate-limit handling, and a per-day spend cap / quota**. Log token usage per job.
- Make generation **idempotent and resumable**, triggered by a scheduler (cron/queue), with a dead-letter path for repeated failures.

## 7. Daily mode vs Infinite mode (exact semantics)

**Daily mode**
- For each *(topic × engine)*, exactly **one** puzzle per calendar day, **the same for everyone worldwide**.
- Selection is deterministic from a **date seed** (define a single global reset time, e.g. 00:00 UTC; document it in the UI). A nightly job finalizes and freezes each daily puzzle into immutable storage *before* the reset.
- Each daily puzzle has its own **global leaderboard** and contributes to the user's **streak/stats**.
- A user may attempt each daily puzzle **once**; record completion, score, time, and guesses. After completion show the result + share card + leaderboard.

**Infinite mode**
- Pull the **next unplayed** valid puzzle for that *(topic × engine)* from the pool (track per-user "seen" set so they don't repeat).
- No global ranked leaderboard (these aren't shared puzzles); instead keep **personal practice stats** (sessions, win rate, best streak, average guesses) and an optional opt-in "infinite points" all-time board.
- Fully **skippable** without penalty; players can chain puzzles indefinitely.
- The **buffer worker** keeps these pools above their low-water-mark (§3).

## 8. Leaderboards & statistics

- **Per-daily-puzzle global leaderboard**, one per *(topic × engine × date)* — this satisfies "one singular board per topic per day, the same for every player in the world." Rank by a defined score (e.g. fewer guesses, then faster solve time; or a points formula per engine). Show top N + the user's own rank.
- **Scoring model:** define per engine. Generally reward correctness, fewer guesses/attempts, and speed; penalize hints used. Keep it explainable.
- **User stats:** current streak, max streak, games played, win %, guess-distribution histograms (Wordle-style), per-topic breakdown.
- **Streaks:** maintained on daily play; define grace rules for missed days.
- **Anti-cheat basics:** validate solutions and timing server-side; the answer never reaches the client before it's needed; rate-limit submissions; sanity-check implausibly fast solves. Don't over-engineer this for v1, but don't trust the client.
- **Share cards:** generate spoiler-free emoji/result share snippets per engine (e.g. the colored-grid pattern for Word Guess).

---

## 9. Skip / anti-boredom mechanics

- **Skip this puzzle** (infinite mode: no penalty, serve the next; daily mode: allow "come back later," but a true skip forfeits that day's entry for that puzzle — make the consequence explicit).
- **Swap topic** instantly from any game screen.
- **"Give me another"** one-tap to pull a fresh infinite puzzle in the same format.
- **Difficulty selector** where the engine supports it.
- A lightweight **"feeling stuck?" hint** (costs leaderboard points) so players progress rather than rage-quit.

## 10. IP, copyright & content safety (hard guardrails)

- **Do not** scrape, bundle, or ship copyrighted media (song clips, movie stills, game art) or trademarked franchise content (specific Pokémon, League of Legends champions, Naruto/One Piece characters, NYT puzzle content, etc.). The famous franchise "-dle" games are mechanical inspiration only.
- Build topics from **generic knowledge domains** (geography, science, sports facts, language) and **openly-licensed / public-domain / open-data** sources (e.g. open country datasets, public reference data, CC0 sources). When Gemini produces facts, **ground and validate** them; never present unverified claims as fact.
- Don't reproduce another product's exact wordmark, logo, or proprietary puzzle archive. Pick your own product name and original visual identity.
- Filter generated content for offensive/unsafe output before it enters the pool.

## 11. Suggested data model (adapt as needed)

- `users` — id, auth, display name, country (optional, for boards), difficulty pref, created_at.
- `user_interests` — user_id, topic_id, weight.
- `topics` — id, name, family, status (draft/approved/live), config (which engines, dataset refs).
- `datasets` — id, topic_id, engine, version, validated_at, payload/ref (entities + attributes, word lists, etc.).
- `puzzles` — id, engine, topic_id, mode (daily/infinite), difficulty, payload (puzzle + solution), validation_status, content_hash, source (gemini/solver), created_at.
- `daily_assignments` — date, topic_id, engine, puzzle_id (immutable once set), reset_at.
- `attempts` — id, user_id, puzzle_id, mode, result, score, num_guesses, duration_ms, hints_used, completed_at.
- `leaderboards` — derived/materialized per (date, topic_id, engine); plus user rank queries.
- `user_stats` — user_id, per-topic + global streaks, histograms, totals.
- `seen_infinite` — user_id, puzzle_id (to avoid repeats).
- `generation_jobs` — id, type, topic_id, engine, status, model, tokens, error, scheduled_for.

## 12. Backend & API sketch

- `POST /onboarding/interests` — save picks.
- `GET /feed` — personalized list of daily cards + infinite entry points.
- `GET /daily/:topic/:engine` — today's frozen puzzle (no solution leak).
- `POST /daily/:puzzleId/attempt` — submit; server validates, scores, returns result + leaderboard slice.
- `GET /infinite/:topic/:engine/next` — next unseen pool puzzle.
- `POST /infinite/:puzzleId/attempt` — practice scoring/stats.
- `GET /leaderboard/:topic/:engine/:date` — global board.
- `GET /stats` — user stats/streaks.
- `POST /skip` — record skip / advance.
- **Admin/worker:** topic approval, dataset build, pool-depth dashboard, manual regen, spend monitor.

Keep solutions server-side; the client receives only what it needs to render the current state.

## 13. Tech stack (suggested, not mandatory)

Pick a stack you can move fast in; a sensible default:
- **Frontend:** React (Next.js) or similar, mobile-first, Tailwind. Clean, fast, snappy puzzle UIs; offline-tolerant where easy.
- **Backend:** Node/TypeScript (or Python) API; a **queue + scheduler** (e.g. cron + a job queue) for generation; a **worker** process for Gemini batch jobs and the buffer refill.
- **Data:** Postgres for durable content/stats; Redis for hot caches (today's puzzles, leaderboards) and rate limiting.
- **Gemini:** official Google Gen AI SDK, server-side, structured output, Batch API, grounding. Verify current model IDs and limits against Google's docs.
- **Auth:** lightweight (email/OAuth); allow guest play with later account linking so onboarding friction is low.

## 14. Build order (ship a vertical slice first)

1. **One engine, end to end:** implement **Word Guess (A)** with a single topic, the validation gate, daily-seed freeze, daily leaderboard, basic stats, and skip. Prove the whole pipeline (generate offline → validate → freeze → serve → score → rank) works.
2. **Add the Gemini generation worker + buffer** for that engine; prove infinite mode never empties and costs nothing per play.
3. **Add the onboarding interest picker** and the personalized feed.
4. **Add engines** in this order: B (attribute deduction — highest payoff for interests), C, then D logic puzzles (with solvers), then E/F/G/H.
5. **Broaden topics** (target 15–25) and the admin topic-approval + dataset-build tooling.
6. **Polish:** share cards, streaks/grace rules, anti-cheat hardening, difficulty tuning, spend dashboards.

## 15. Definition of done (acceptance criteria)

- A new user completes onboarding and immediately sees a personalized feed.
- Daily mode: two different users, anywhere, on the same date get the **identical** puzzle per *(topic × engine)*, share **one** global leaderboard per topic per day, and accrue streaks.
- Infinite mode: a user can play many puzzles back-to-back in one sitting without repeats or running out; **zero live Gemini calls** on the play path.
- Every served puzzle has passed the validation gate; logic puzzles are provably unique-solution; no puzzle is unsolvable.
- Gemini runs only in background jobs, behind a spend cap, with the API key never exposed to clients.
- No copyrighted media or trademarked franchise content ships in the app.
- Skipping and topic-swapping work everywhere; nothing dead-ends the player.
